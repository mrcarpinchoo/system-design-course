Why Replication?¿Por qué Replicar?

Replication keeps copies of data on multiple machines to improve reliability and performance.La replicación mantiene copias de los datos en múltiples máquinas para mejorar la confiabilidad y el rendimiento.

DurabilityDurabilidad

Survive hardware failures without data lossSobrevivir fallas de hardware sin perder datos

If one node fails, another replica holds the data. Cloud providers replicate across availability zones to protect against datacenter outages (e.g., AWS S3 stores 3+ copies across AZs).Si un nodo falla, otra réplica conserva los datos. Los proveedores cloud replican entre zonas de disponibilidad para proteger contra caídas de centros de datos (ej., AWS S3 almacena 3+ copias entre AZs).

AvailabilityDisponibilidad

Automatic failover keeps the system runningEl failover automático mantiene el sistema operando

When the primary node becomes unreachable, a replica is promoted to take over. This enables high availability (99.99%+) without manual intervention.Cuando el nodo primario queda inaccesible, una réplica se promueve para tomar el control. Esto permite alta disponibilidad (99.99%+) sin intervención manual.

LatencyLatencia

Clients read from the nearest replicaLos clientes leen de la réplica más cercana

Geo-distributed replicas reduce round-trip time. A user in Tokyo reads from a local replica instead of crossing the Pacific to a US datacenter (200ms → 5ms).Las réplicas geo-distribuidas reducen el tiempo de ida y vuelta. Un usuario en Tokio lee de una réplica local en vez de cruzar el Pacífico hacia un datacenter en EE.UU. (200ms → 5ms).

ThroughputRendimiento

Distribute reads across multiple replicasDistribuir lecturas entre múltiples réplicas

Instead of one node handling all queries, read traffic is spread across N replicas, increasing total read capacity by Nx. Write capacity is still limited to the leader node(s).En lugar de que un solo nodo maneje todas las consultas, el tráfico de lectura se distribuye entre N réplicas, aumentando la capacidad de lectura Nx. La capacidad de escritura sigue limitada al nodo(s) líder.

Leader-Follower ReplicationReplicación Líder-Seguidor

One node (leader) accepts all writes and propagates changes to followers. Clients can read from any replica.Un nodo (líder) acepta todas las escrituras y propaga los cambios a los seguidores. Los clientes pueden leer de cualquier réplica.

BenefitsBeneficios

Simple, no write conflicts, easy to reason aboutSimple, sin conflictos de escritura, fácil de razonar

All writes go through a single leader, so there are no conflicts. Read replicas scale read throughput linearly. Most relational databases support this pattern natively (PostgreSQL, MySQL, SQL Server).Todas las escrituras pasan por un solo líder, por lo que no hay conflictos. Las réplicas de lectura escalan el rendimiento linealmente. La mayoría de bases de datos relacionales soportan este patrón nativamente (PostgreSQL, MySQL, SQL Server).

TradeoffsCompromisos

Leader is a single point of failure for writesEl líder es un punto único de falla para escrituras

If the leader goes down, writes are blocked until failover completes. Followers may serve stale data (replication lag). Failover can cause data loss if the leader had unreplicated writes.Si el líder cae, las escrituras se bloquean hasta que el failover se complete. Los seguidores pueden servir datos obsoletos (lag de replicación). El failover puede causar pérdida de datos si el líder tenía escrituras no replicadas.

Sync vs Async ReplicationReplicación Síncrona vs Asíncrona

Should the leader wait for followers to confirm before acknowledging the write to the client?¿Debe el líder esperar la confirmación de los seguidores antes de responder al cliente?

Multi-Leader ReplicationReplicación Multi-Líder

Multiple nodes accept writes and replicate changes to each other. Enables geo-distributed writes but introduces conflicts.Múltiples nodos aceptan escrituras y replican cambios entre sí. Permite escrituras geo-distribuidas pero introduce conflictos.

Conflict ResolutionResolución de Conflictos

When two leaders modify the same data concurrently, a conflict arises. Strategies include last-write-wins (LWW), version vectors, and application-level merge logic.Cuando dos líderes modifican los mismos datos concurrentemente, surge un conflicto. Las estrategias incluyen last-write-wins (LWW), vectores de versión y lógica de merge a nivel de aplicación.

LWW is simple but can silently drop writes. Version vectors detect conflicts accurately but require application logic to resolve them. CRDTs (Conflict-free Replicated Data Types) offer automatic resolution for certain data structures like counters and sets.LWW es simple pero puede descartar escrituras silenciosamente. Los vectores de versión detectan conflictos con precisión pero requieren lógica de aplicación para resolverlos. Los CRDTs ofrecen resolución automática para ciertas estructuras de datos como contadores y conjuntos.

Leaderless ReplicationReplicación sin Líder

The client writes to multiple nodes simultaneously. Quorum rules determine success: W + R > N ensures overlap.El cliente escribe a múltiples nodos simultáneamente. Reglas de quórum determinan el éxito: W + R > N asegura traslape.

How Quorums WorkCómo Funcionan los Quórums

With N replicas, writes need W acknowledgments and reads query R nodes. If W + R > N, at least one node has the latest value.Con N réplicas, las escrituras necesitan W confirmaciones y las lecturas consultan R nodos. Si W + R > N, al menos un nodo tiene el valor más reciente.

Common configurations: N=3, W=2, R=2 (balanced). N=3, W=3, R=1 (strong write, fast reads). N=3, W=1, R=3 (fast writes, strong reads). Used by Dynamo, Cassandra, and Riak. Sloppy quorums relax rules for availability during partitions.Configuraciones comunes: N=3, W=2, R=2 (balanceado). N=3, W=3, R=1 (escritura fuerte, lecturas rápidas). N=3, W=1, R=3 (escrituras rápidas, lecturas fuertes). Usado por Dynamo, Cassandra y Riak. Los quórums laxos relajan las reglas para disponibilidad durante particiones.

Conflict ResolutionResolución de Conflictos

When concurrent writes happen on different replicas, the system must decide which value wins.Cuando escrituras concurrentes ocurren en diferentes réplicas, el sistema debe decidir qué valor prevalece.

Last-Write-Wins (LWW)Last-Write-Wins (LWW)

Timestamp determines the winner; simple but lossyLa marca de tiempo determina al ganador; simple pero con pérdida

Each write is tagged with a timestamp. The write with the highest timestamp wins. Problem: clock skew between nodes can cause the “wrong” write to win. Used by Cassandra as default conflict resolution strategy.Cada escritura se etiqueta con una marca de tiempo. La escritura con la marca más alta gana. Problema: el desfase de relojes entre nodos puede hacer que la escritura “incorrecta” gane. Usado por Cassandra como estrategia predeterminada.

Version VectorsVectores de Versión

Track causal history per replica to detect conflictsRastrean historial causal por réplica para detectar conflictos

Each node maintains a vector of counters (one per replica). Two versions are concurrent if neither dominates the other. The application is then asked to merge the conflicting values. Used by Amazon DynamoDB and Riak.Cada nodo mantiene un vector de contadores (uno por réplica). Dos versiones son concurrentes si ninguna domina a la otra. La aplicación entonces debe fusionar los valores en conflicto. Usado por Amazon DynamoDB y Riak.

Application-Level MergeMerge a Nivel de Aplicación

Custom logic to merge conflicting valuesLógica personalizada para fusionar valores en conflicto

The application provides a merge function invoked when conflicts are detected. Example: a shopping cart merges items by taking the union of both carts. Offers maximum flexibility but adds complexity.La aplicación proporciona una función de merge que se invoca al detectar conflictos. Ejemplo: un carrito de compras fusiona artículos tomando la unión de ambos carritos. Ofrece máxima flexibilidad pero agrega complejidad.

CRDTsCRDTs

Data structures that auto-merge without conflictsEstructuras de datos que se auto-fusionan sin conflictos

Conflict-free Replicated Data Types (CRDTs) are mathematically guaranteed to converge. Examples: G-Counter (grow-only counter), OR-Set (observed-remove set), LWW-Register. Used by Redis CRDT, Automerge, and Yjs for collaborative editing.Los tipos de datos replicados sin conflicto (CRDTs) están matemáticamente garantizados para converger. Ejemplos: G-Counter (contador de solo crecimiento), OR-Set (conjunto con eliminación observada), LWW-Register. Usados por Redis CRDT, Automerge y Yjs para edición colaborativa.

Consistency ModelsModelos de Consistencia

Different applications require different guarantees about when replicas reflect the latest write.Diferentes aplicaciones requieren diferentes garantías sobre cuándo las réplicas reflejan la última escritura.

Strong ConsistencyConsistencia Fuerte

Linearizable — reads always return the latest writeLinealizable — las lecturas siempre devuelven la última escritura

After a write completes, all subsequent reads (from any client, any replica) see the new value. Requires synchronous replication or consensus protocols (Raft, Paxos). High latency cost. Used by: Google Spanner, CockroachDB, etcd, ZooKeeper.Después de que una escritura se completa, todas las lecturas subsecuentes (de cualquier cliente, cualquier réplica) ven el nuevo valor. Requiere replicación síncrona o protocolos de consenso (Raft, Paxos). Alto costo de latencia. Usado por: Google Spanner, CockroachDB, etcd, ZooKeeper.

Eventual ConsistencyConsistencia Eventual

Replicas converge — given enough time, all return same valueLas réplicas convergen — dado suficiente tiempo, todas devuelven el mismo valor

After a write, replicas may temporarily return stale data. The system guarantees that, if no new writes occur, all replicas will eventually return the same value. Low latency, high availability. Used by: DynamoDB, Cassandra, DNS, S3.Después de una escritura, las réplicas pueden devolver datos obsoletos temporalmente. El sistema garantiza que, si no hay nuevas escrituras, todas las réplicas eventualmente devolverán el mismo valor. Baja latencia, alta disponibilidad. Usado por: DynamoDB, Cassandra, DNS, S3.

Causal ConsistencyConsistencia Causal

Preserves cause-and-effect order between related operationsPreserva el orden causa-efecto entre operaciones relacionadas

If operation A causally precedes operation B, all replicas see A before B. Concurrent (unrelated) operations may appear in any order. Weaker than strong consistency but stronger than eventual. Used by: MongoDB (causal sessions), COPS, Eiger.Si la operación A precede causalmente a la operación B, todas las réplicas ven A antes que B. Las operaciones concurrentes (no relacionadas) pueden aparecer en cualquier orden. Más débil que la consistencia fuerte pero más fuerte que la eventual. Usado por: MongoDB (sesiones causales), COPS, Eiger.

Cloud Storage ServicesServicios de Almacenamiento en la Nube

Major cloud providers offer managed replication for storage and database services across regions.Los principales proveedores cloud ofrecen replicación administrada para servicios de almacenamiento y bases de datos entre regiones.

AWS S3 / EBSAWS S3 / EBS

Cross-Region Replication (CRR) for S3; multi-AZ EBS snapshotsReplicación entre regiones (CRR) para S3; snapshots multi-AZ de EBS

S3 stores objects across 3+ AZs by default (11 nines durability). CRR asynchronously copies objects to a bucket in another region for disaster recovery. EBS volumes can be replicated via snapshots stored in S3. RDS supports read replicas across AZs and regions.S3 almacena objetos en 3+ AZs por defecto (11 nueves de durabilidad). CRR copia objetos asíncronamente a un bucket en otra región para recuperación ante desastres. Los volúmenes EBS pueden replicarse vía snapshots almacenados en S3. RDS soporta réplicas de lectura entre AZs y regiones.

Azure Blob StorageAzure Blob Storage

Geo-redundant storage (GRS) replicates to a paired regionAlmacenamiento geo-redundante (GRS) replica a una región emparejada

Azure offers LRS (3 copies in one datacenter), ZRS (3 AZs), GRS (cross-region pair), and GZRS (ZRS + cross-region). Read-access GRS (RA-GRS) allows reads from the secondary region. Cosmos DB offers 5 consistency levels from strong to eventual.Azure ofrece LRS (3 copias en un datacenter), ZRS (3 AZs), GRS (par entre regiones) y GZRS (ZRS + entre regiones). RA-GRS permite lecturas desde la región secundaria. Cosmos DB ofrece 5 niveles de consistencia desde fuerte hasta eventual.

GCP Cloud StorageGCP Cloud Storage

Multi-region buckets replicate across geographies automaticallyLos buckets multi-región replican automáticamente entre geografías

GCP Cloud Storage offers regional, dual-region, and multi-region buckets. Multi-region provides geo-redundancy with strong consistency. Cloud Spanner is the only globally distributed database offering external consistency (stronger than linearizability).GCP Cloud Storage ofrece buckets regionales, de doble región y multi-región. Multi-región proporciona geo-redundancia con consistencia fuerte. Cloud Spanner es la única base de datos globalmente distribuida que ofrece consistencia externa (más fuerte que la linealizabilidad).

OCI Object StorageOCI Object Storage

Cross-region replication with configurable policiesReplicación entre regiones con políticas configurables

Oracle Cloud Infrastructure Object Storage replicates across fault domains within an AD and across ADs within a region. Cross-region replication is available via replication policies. Autonomous Database supports Data Guard for automated failover and read replicas.Oracle Cloud Infrastructure Object Storage replica entre dominios de falla dentro de un AD y entre ADs dentro de una región. La replicación entre regiones está disponible vía políticas. Autonomous Database soporta Data Guard para failover automatizado y réplicas de lectura.

SummaryResumen

Replication StrategiesEstrategias de Replicación

Leader-follower is the most common. Multi-leader enables geo-writes. Leaderless uses quorums for high availability.Líder-seguidor es la más común. Multi-líder permite escrituras geo-distribuidas. Sin líder usa quórums para alta disponibilidad.

Consistency TradeoffsCompromisos de Consistencia

Strong consistency has high latency. Eventual consistency is fast but stale. Causal consistency is a practical middle ground.Consistencia fuerte tiene alta latencia. Consistencia eventual es rápida pero con datos obsoletos. Consistencia causal es un punto medio práctico.
