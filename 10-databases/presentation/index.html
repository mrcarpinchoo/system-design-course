CAP TheoremTeorema CAP

CAP Theorem ExplainedTeorema CAP Explicado

In a distributed system, you can only guarantee two of three properties En un sistema distribuido, solo puedes garantizar dos de tres propiedades

CAP PropertiesPropiedades CAP

Consistency

Every read receives the most recent write or an error. All nodes see the same data at the same time.Cada lectura recibe la escritura más reciente o un error. Todos los nodos ven los mismos datos al mismo tiempo.

Example: After a write, every subsequent read returns that value. Like a whiteboard everyone can see.Ejemplo: Después de una escritura, toda lectura posterior retorna ese valor. Como un pizarrón visible para todos.

Availability

Every request receives a (non-error) response, without guarantee it contains the most recent write.Cada solicitud recibe una respuesta (sin error), sin garantía de que contenga la escritura más reciente.

Example: Every request gets a response. Like a 24/7 store — always open.Ejemplo: Cada solicitud recibe respuesta. Como una tienda 24/7 — siempre abierta.

Partition Tolerance

The system continues to operate despite network partitions between nodes.El sistema sigue funcionando a pesar de particiones de red entre nodos.

Example: The system works even if network cables between servers are cut.Ejemplo: El sistema funciona incluso si se cortan los cables de red entre servidores.

CAP CombinationsCombinaciones CAP

CP — Consistent + Partition TolerantConsistente + Tolerante a Particiones

Sacrifices availability. Waits for consensus.Sacrifica disponibilidad. Espera consenso.

MongoDB, HBase, Redis Cluster

MongoDB uses replica sets with primary election. During a partition, minority nodes become unavailable. HBase uses ZooKeeper for consensus — if partition splits the quorum, writes block.MongoDB usa conjuntos de réplicas con elección de primario. Durante una partición, los nodos en minoría se vuelven no disponibles. HBase usa ZooKeeper para consenso — si la partición divide el quórum, las escrituras se bloquean.

AP — Available + Partition TolerantDisponible + Tolerante a Particiones

Sacrifices consistency. Returns stale data.Sacrifica consistencia. Retorna datos obsoletos.

DynamoDB, Cassandra, CouchDB

DynamoDB uses eventual consistency by default — writes propagate asynchronously. Cassandra uses gossip protocol — all nodes accept writes, conflicts resolved by timestamp.DynamoDB usa consistencia eventual por defecto — las escrituras se propagan asíncronamente. Cassandra usa protocolo gossip — todos los nodos aceptan escrituras, conflictos resueltos por timestamp.

CA — Consistent + AvailableConsistente + Disponible

No partition tolerance. Single-node only.Sin tolerancia a particiones. Solo un nodo.

PostgreSQL, MySQL (single)

Only possible in single-node systems (no network partition possible). Traditional PostgreSQL or MySQL running on one server. Not viable for distributed systems.Solo posible en sistemas de un solo nodo (sin partición de red posible). PostgreSQL o MySQL tradicional ejecutándose en un servidor. No viable para sistemas distribuidos.

ACID

A — Atomicity

All or nothing. Transaction completes fully or rolls back entirely.Todo o nada. La transacción se completa o se revierte por completo.

Like an ATM transaction — either the money is transferred OR nothing happens.Como una transacción en cajero — o el dinero se transfiere O no pasa nada.

C — Consistency

Data moves from one valid state to another. Constraints enforced.Los datos pasan de un estado válido a otro. Se aplican restricciones.

Like a bank balance — after any transfer, total money in the system stays the same.Como un saldo bancario — después de cualquier transferencia, el total del sistema se mantiene.

I — Isolation

Concurrent transactions don’t interfere with each other.Las transacciones concurrentes no interfieren entre sí.

Like two people editing a shared doc — they don't see each other's half-written changes.Como dos personas editando un documento compartido — no ven los cambios a medias del otro.

D — Durability

Once committed, data survives crashes and power failures.Una vez confirmados, los datos sobreviven a fallos y cortes de energía.

Like writing to a notebook — once you write it, it's there even if you close the book.Como escribir en un cuaderno — una vez escrito, permanece aunque cierres el libro.

Database IndexingIndexación de Bases de Datos

Indexing — Key InsightIndexación — Punto Clave

Indexes trade write speed for read speed. Like a book index — without it you'd scan every page. B-tree indexes are default in most RDBMS. Composite indexes follow leftmost prefix rule.Los índices intercambian velocidad de escritura por velocidad de lectura. Como el índice de un libro — sin él tendrías que leer cada página. Los índices B-tree son predeterminados. Los índices compuestos siguen la regla del prefijo izquierdo.

Materialized ViewsVistas Materializadas

Materialized Views — Key InsightVistas Materializadas — Punto Clave

Pre-computed query results stored as a table. Trade storage for speed. Must be refreshed — stale data risk. PostgreSQL supports REFRESH MATERIALIZED VIEW CONCURRENTLY.Resultados de consulta pre-computados almacenados como tabla. Intercambian almacenamiento por velocidad. Deben refrescarse — riesgo de datos obsoletos. PostgreSQL soporta REFRESH MATERIALIZED VIEW CONCURRENTLY.

Database CachingCaché de Base de Datos

Cache hit ratio above 95% is the target. Cache invalidation is one of the two hard problems in CS. Database ReplicationReplicación de Base de Datos

Replication — Key InsightReplicación — Punto Clave

Replication lag is the key metric — time between leader write and follower applying it. Synchronous replication guarantees zero lag but hurts write performance. Most systems use asynchronous.El retraso de replicación es la métrica clave — tiempo entre escritura del líder y aplicación en el seguidor. La replicación síncrona garantiza cero retraso pero afecta escrituras. La mayoría usa asíncrona.

Choose shard key wisely — a bad key creates hotspots. User ID is common but risky if one user generates disproportionate traffic. Hash-based sharding distributes evenly but makes range queries hard.Elige la clave de shard sabiamente — una mala clave crea hotspots. User ID es común pero riesgoso si un usuario genera tráfico desproporcionado. Sharding basado en hash distribuye uniformemente pero dificulta consultas por rango.

Scale RequirementsRequisitos de Escala

Predictable growth? SQL + vertical. Massive scale? NoSQL + sharding.¿Crecimiento predecible? SQL + vertical. ¿Escala masiva? NoSQL + sharding.

If you expect <10TB and <10K requests/second, SQL with read replicas is likely sufficient. Beyond that, consider DynamoDB or Cassandra for horizontal scaling.Si esperas <10TB y <10K solicitudes/segundo, SQL con réplicas de lectura probablemente sea suficiente. Consistency NeedsNecesidades de Consistencia

Strong ACID? RDS. Eventual OK? DynamoDB. Tunable? Cassandra.¿ACID fuerte? RDS. ¿Eventual OK? DynamoDB. ¿Tunable? Cassandra.

Financial transactions demand strong consistency. User profiles and social feeds tolerate eventual. Cassandra lets you tune per-query with consistency levels (ONE, QUORUM, ALL).Las transacciones financieras demandan consistencia fuerte. Perfiles de usuario y feeds sociales toleran eventual. SQL is better when queries are ad-hoc or evolving.Mapea todos tus patrones de acceso antes de elegir. DynamoDB es óptimo cuando conoces tus consultas de antemano. SQL es mejor cuando las consultas son ad-hoc o evolucionan.
