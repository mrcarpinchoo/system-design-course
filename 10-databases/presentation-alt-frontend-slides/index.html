SQL vs NoSQL

SQL (Relational)

  • Structured schema with tables and rows
  • ACID compliant transactions
  • Complex JOIN queries supported
  • Vertical scaling (scale up)
  • Strong consistency guarantees
  • MySQL, PostgreSQL, Oracle

NoSQL (Non-Relational)

  • Flexible schema or schema-less
  • BASE model with eventual consistency
  • Optimized for specific data patterns
  • Horizontal scaling (scale out)
  • High throughput and low latency
  • MongoDB, DynamoDB, Cassandra

4 / 25

CAP Properties

C

Consistency

Every read returns the most recent write. All nodes see the same data at the same time.

A

Availability

Every request receives a response (success or failure), even if some nodes are down.

P

Partition Tolerance

System continues operating despite network partitions between nodes.

5 / 25

CAP Combinations

CP

Consistency + Partition Tolerance

MongoDB, HBase, Redis

Sacrifices availability during partitions

AP

Availability + Partition Tolerance

Cassandra, DynamoDB, CouchDB

Sacrifices consistency (eventual)

CA

Consistency + Availability

PostgreSQL, MySQL (single-node)

Only possible without network partitions

ACID Properties

A

Atomicity

All operations in a transaction succeed or all fail. No partial updates.

C

Consistency

A transaction brings the database from one valid state to another.

I

Isolation

Concurrent transactions do not interfere with each other.

D

Durability

Committed transactions persist even after system failures.

8 / 25

BASE Properties

BA

Basically Available

The system guarantees availability per the CAP theorem. Partial failures do not cause total system failure.

S

Soft State

The system state may change over time, even without new input, due to eventual consistency propagation.

E

Eventually Consistent

The system will become consistent over time, given no new updates. Reads may return stale data temporarily.

20 / 25

Read Consistency

Eventually Consistent

  • Response may not reflect latest write
  • Higher throughput, lower latency
  • Default for DynamoDB reads
  • Data converges within seconds
  • Cost: 0.5 RCU per 4 KB

Strongly Consistent

  • Always returns most recent write
  • Higher latency, lower throughput
  • Must be explicitly requested
  • May be unavailable during partitions
  • Cost: 1 RCU per 4 KB

Key Takeaways

📚

Know Your Data Model

SQL for structured relationships, NoSQL for flexible schemas. CAP theorem guides your tradeoffs.

📈

Scale Progressively

Start with indexing and caching. Add replication for read scaling. Shard when you outgrow a single node.

☁

Leverage Managed Services

AWS RDS and DynamoDB handle the operational burden so you can focus on application logic.
