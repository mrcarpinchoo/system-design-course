Why Authorization Code Flow?¿Por qué Authorization Code Flow?

Most secure OAuth flow for server-side apps. The code exchange happens server-to-server.Flujo OAuth más seguro para apps del lado servidor. El intercambio de código ocurre servidor a servidor.

  • Authorization code is short-lived and single-useEl código de autorización es de corta vida y un solo uso

SSO & SAMLSSO y SAML

Single Sign-On allows users to authenticate once and access multiple applications. SAML is the enterprise standard for SSO.Single Sign-On permite a los usuarios autenticarse una vez y acceder a múltiples aplicaciones. SAML es el estándar empresarial para SSO.

SAML vs OAuth 2.0 / OIDCSAML vs OAuth 2.0 / OIDC

SAML uses XML assertions for enterprise SSO. OAuth 2.0 + OIDC uses JSON tokens and is preferred for modern web/mobile apps.SAML usa aserciones XML para SSO empresarial. OAuth 2.0 + OIDC usa tokens JSON y es preferido para apps web/móvil modernas.

  • SAML: XML-based, heavyweight, enterprise legacy systemsSAML: basado en XML, pesado, sistemas legados empresariales
  • OIDC: JSON-based, lightweight, modern APIs and SPAsOIDC: basado en JSON, ligero, APIs modernas y SPAs
  • Both support SSO but with different protocolsAmbos soportan SSO pero con diferentes protocolos

SummaryResumen

Identity & AccessIdentidad y Acceso

Authentication (AuthN) verifies identity; Authorization (AuthZ) controls access. Sessions are stateful; JWTs are stateless. PASETO improves on JWT security by eliminating algorithm negotiation.Autenticación (AuthN) verifica identidad; Autorización (AuthZ) controla acceso. Las sesiones son con estado; los JWTs son sin estado. PASETO mejora la seguridad de JWT al eliminar la negociación de algoritmos.

Protocols & FlowsProtocolos y Flujos

OAuth 2.0 Authorization Code flow is the standard for secure delegation. SAML handles enterprise SSO. OIDC adds identity on top of OAuth. Use PKCE for public clients.El flujo OAuth 2.0 Authorization Code es el estándar para delegación segura. SAML maneja SSO empresarial. OIDC agrega identidad sobre OAuth. Usar PKCE para clientes públicos.

No single mechanism is sufficient.Capas de múltiples controles de seguridad: HTTPS, cookies seguras, MFA, rotación de tokens, rate limiting, validación de entrada y mínimo privilegio. Ningún mecanismo único es suficiente.
