Why Caching?¿Por qué Caché?

Every request needs to be fast. Caching reduces load on databases and cuts latency.Cada solicitud necesita ser rápida. El caché reduce la carga en bases de datos y disminuye la latencia.

<1ms Cache hit latency Latencia de acierto en caché
~100ms Database query latency Latencia de consulta a BD
10x Throughput improvement Mejora en rendimiento

Why does it matter?¿Por qué importa?

Caching creates breathing room for slower backend systems.El caché crea espacio para sistemas backend más lentos.

By storing frequently accessed data closer to the consumer, caches reduce redundant database queries, lower network round-trips, and absorb traffic spikes that would otherwise overwhelm the origin server.Al almacenar datos frecuentemente accedidos más cerca del consumidor, los cachés reducen consultas redundantes a la base de datos, disminuyen viajes de red y absorben picos de tráfico que de otra forma abrumarían al servidor origen.

What is Distributed Caching?¿Qué es el Caché Distribuido?

A distributed cache spreads data across multiple nodes, providing shared access for all application servers with high availability and horizontal scalability.Un caché distribuido reparte datos entre múltiples nodos, proporcionando acceso compartido para todos los servidores de aplicación con alta disponibilidad y escalabilidad horizontal.

Write-Through CachingCaché Write-Through

Every write goes through the cache before hitting the database. The cache is always consistent with the source of truth.Cada escritura pasa por el caché antes de llegar a la base de datos. El caché siempre es consistente con la fuente de verdad.

BenefitsBeneficios

Strong consistency, always trust the cacheConsistencia fuerte, siempre confía en el caché

The cache always has the latest data because every write updates it first. Read operations never return stale data. This simplifies application logic since there is no need for cache invalidation strategies.El caché siempre tiene los datos más recientes porque cada escritura lo actualiza primero. Las operaciones de lectura nunca devuelven datos obsoletos. Esto simplifica la lógica de la aplicación ya que no se necesitan estrategias de invalidación.

Trade-offsCompensaciones

Slower writes, write amplificationEscrituras lentas, amplificación de escritura

Every write must complete in two systems (cache + database) before returning success. This doubles write latency and creates a bottleneck if the database is slow. Also causes write amplification for data that is rarely read.Cada escritura debe completarse en dos sistemas (caché + base de datos) antes de retornar éxito. Esto duplica la latencia de escritura y crea un cuello de botella si la base de datos es lenta. También causa amplificación de escritura para datos que rara vez se leen.

Cache-Aside (Lazy Loading)Cache-Aside (Carga Perezosa)

The most common approach. Writes go directly to the database; the cache loads data on read miss.El enfoque más común. Las escrituras van directo a la base de datos; el caché carga datos en caso de fallo de lectura.

BenefitsBeneficios

Reduces cache pollution, lower write load on cacheReduce la contaminación del caché, menor carga de escritura

Only data that is actually read gets cached, avoiding storage waste on rarely accessed items. Writes are fast because they only hit the database. Good for read-heavy workloads where not all data needs to be cached.Solo los datos que realmente se leen se almacenan en caché, evitando desperdiciar almacenamiento en elementos poco accedidos. Las escrituras son rápidas porque solo llegan a la base de datos. Bueno para cargas de trabajo intensivas en lectura.

Trade-offsCompensaciones

First read causes a miss, stale data windowPrimera lectura causa un fallo, ventana de datos obsoletos

Cold start problem: the first request for any key always hits the database. After a write, the cache entry is deleted, so the next read will have increased latency. There is a brief window where the cache may serve stale data if invalidation is delayed.Problema de arranque en frío: la primera solicitud para cualquier clave siempre llega a la base de datos. Después de una escritura, la entrada del caché se elimina, por lo que la siguiente lectura tendrá mayor latencia.

Write-Back CachingCaché Write-Back

The cache becomes the primary write target. The database lags behind, receiving batched writes asynchronously.El caché se convierte en el destino principal de escritura. La base de datos queda rezagada, recibiendo escrituras por lotes de forma asíncrona.

BenefitsBeneficios

Fastest write latency, reduces DB write loadMenor latencia de escritura, reduce carga de escritura en BD

Writes return immediately after updating the cache, without waiting for the database. Multiple writes to the same key can be coalesced into a single database write. Ideal for bursty traffic patterns like game state updates or real-time analytics counters.Las escrituras retornan inmediatamente después de actualizar el caché, sin esperar a la base de datos. Múltiples escrituras a la misma clave pueden consolidarse en una sola escritura a la BD. Ideal para patrones de tráfico en ráfagas como actualizaciones de estado de juegos o contadores de analítica.

Trade-offsCompensaciones

Risk of data loss, harder consistency guaranteesRiesgo de pérdida de datos, garantías de consistencia más difíciles

If the cache crashes before flushing to the database, uncommitted writes are lost. Requires cache replication or write-ahead logging to mitigate. The database may serve stale data if read directly. Debugging is harder because the source of truth is split between cache and database.Si el caché falla antes de vaciar a la base de datos, las escrituras no confirmadas se pierden. Requiere replicación del caché o registro de escritura anticipada. La base de datos puede servir datos obsoletos si se lee directamente.

Cache Consistency IssuesProblemas de Consistencia del Caché

Distributed caches introduce coordination problems that must be addressed at the application or infrastructure level.Los cachés distribuidos introducen problemas de coordinación que deben abordarse a nivel de aplicación o infraestructura.

Stale ReadsLecturas Obsoletas

Client reads from a cache node not yet updatedEl cliente lee de un nodo de caché aún no actualizado

After a write, other cache nodes may still hold the old value until invalidation propagates. This window can range from milliseconds to seconds depending on the invalidation mechanism.Después de una escritura, otros nodos de caché pueden aún contener el valor anterior hasta que se propague la invalidación. Esta ventana puede ir de milisegundos a segundos.

Concurrent UpdatesActualizaciones Concurrentes

Two services write to the same key from different nodesDos servicios escriben la misma clave desde diferentes nodos

Without distributed locking or optimistic concurrency control, the last writer wins. This can lead to lost updates where one service overwrites changes made by another. Solutions include versioned keys and compare-and-swap operations.Sin bloqueo distribuido o control de concurrencia optimista, gana el último escritor. Esto puede llevar a actualizaciones perdidas. Las soluciones incluyen claves versionadas y operaciones compare-and-swap.

Partial InvalidationInvalidación Parcial

Service updates DB but fails to invalidate all cache nodesEl servicio actualiza BD pero falla al invalidar todos los nodos

Network partitions or service failures can prevent invalidation messages from reaching all cache nodes. Some nodes serve stale data while others are current. TTL-based expiration provides a safety net for eventual consistency.Particiones de red o fallas de servicio pueden impedir que los mensajes de invalidación lleguen a todos los nodos. Algunos nodos sirven datos obsoletos mientras otros están actualizados. La expiración basada en TTL proporciona una red de seguridad.

Network DelaysRetrasos de Red

Pub/sub message meant to trigger cache update gets lostMensaje pub/sub para actualizar caché se pierde

Event-driven invalidation relies on message delivery guarantees. If the message broker drops a message or the subscriber is temporarily down, the cache remains stale. Combine with TTL expiration as a fallback mechanism.La invalidación basada en eventos depende de garantías de entrega de mensajes. Si el broker pierde un mensaje o el suscriptor está temporalmente caído, el caché permanece obsoleto. Combina con expiración TTL como mecanismo de respaldo.

Thundering Herd ProblemProblema de Estampida

When a popular cache entry expires, thousands of clients simultaneously hammer the backend database.Cuando una entrada popular del caché expira, miles de clientes golpean simultáneamente la base de datos backend.

SolutionsSoluciones

Request coalescing, staggered TTLs, probabilistic early refreshCoalescencia de solicitudes, TTLs escalonados, refresco temprano probabilístico

Request coalescing (singleflight): only one goroutine fetches from the database; others wait for its result. Staggered TTLs: add random jitter to expiration times so keys do not expire simultaneously. Probabilistic early refresh: refresh the key before expiration based on a probability that increases as TTL approaches zero.Coalescencia de solicitudes: solo una goroutine consulta la BD; las demás esperan su resultado. TTLs escalonados: agregar jitter aleatorio a los tiempos de expiración para que las claves no expiren simultáneamente. Refresco temprano probabilístico: refrescar la clave antes de la expiración basado en una probabilidad que aumenta conforme el TTL se acerca a cero.

Eviction PoliciesPolíticas de Desalojo

When cache memory is full, eviction policies decide which entries to remove to make room for new data.Cuando la memoria del caché está llena, las políticas de desalojo deciden qué entradas eliminar para hacer espacio a nuevos datos.

LRU (Least Recently Used)

Evicts the item not accessed for the longest timeDesaloja el elemento no accedido por más tiempo

The most common default policy. Maintains a doubly-linked list ordered by access time. Works well for most workloads because recently accessed data is likely to be accessed again. Redis uses an approximated LRU with sampling for efficiency.La política predeterminada más común. Mantiene una lista doblemente enlazada ordenada por tiempo de acceso. Funciona bien para la mayoría de las cargas de trabajo. Redis usa un LRU aproximado con muestreo para eficiencia.

LFU (Least Frequently Used)

Evicts the item accessed fewest times overallDesaloja el elemento accedido menos veces en total

Better for skewed access distributions where a small set of keys receives the majority of requests. Tracks access frequency with counters. Drawback: new items start with low frequency and may be evicted prematurely. Redis LFU uses a logarithmic counter with decay to handle this.Mejor para distribuciones de acceso sesgadas donde un pequeño conjunto de claves recibe la mayoría de las solicitudes. Rastrea la frecuencia de acceso con contadores. Redis LFU usa un contador logarítmico con decaimiento.

FIFO (First In, First Out)

Evicts the oldest entries regardless of access patternsDesaloja las entradas más antiguas sin importar patrones de acceso

The simplest eviction strategy. Uses a queue: first item in is the first to be evicted. Does not consider access frequency or recency, which makes it suboptimal for most real-world workloads. Useful when entries have uniform access probability or as a baseline for comparison.La estrategia de desalojo más simple. Usa una cola: el primer elemento en entrar es el primero en salir. No considera frecuencia ni recencia de acceso, lo que la hace suboptimal para la mayoría de las cargas de trabajo reales.

TTL (Time to Live)

Entries expire after a fixed duration automaticallyLas entradas expiran automáticamente después de una duración fija

Not a traditional eviction policy, but essential for cache freshness. Each key gets an expiration timestamp. After the TTL elapses, the key is removed (or lazily cleaned up on next access). Combine with LRU or LFU for a robust eviction strategy. Short TTLs improve freshness but increase cache misses.No es una política de desalojo tradicional, pero es esencial para la frescura del caché. Cada clave obtiene una marca de tiempo de expiración. Combina con LRU o LFU para una estrategia de desalojo robusta. TTLs cortos mejoran la frescura pero aumentan los fallos de caché.

Redis vs Memcached

Redis

  • Rich data structures (lists, sets, hashes, sorted sets)Estructuras de datos ricas (listas, conjuntos, hashes, sorted sets)
  • Persistence (RDB snapshots + AOF)Persistencia (snapshots RDB + AOF)
  • Replication + Redis Cluster for shardingReplicación + Redis Cluster para sharding
  • Lua scripting for atomic operationsScripting Lua para operaciones atómicas
  • Pub/Sub messaging built-inMensajería pub/sub integrada
  • Single-threaded event loopBucle de eventos de un solo hilo

Best for: session stores, leaderboards, rate limiting, real-time analytics, message queues, and any use case requiring complex data manipulation. The rich data types eliminate the need for serialization/deserialization in many scenarios.Ideal para: almacén de sesiones, tablas de clasificación, limitación de tasa, analítica en tiempo real, colas de mensajes y cualquier caso que requiera manipulación compleja de datos.

Memcached

  • Simple key-value storage onlyAlmacenamiento simple de clave-valor únicamente
  • Multi-threaded architectureArquitectura multi-hilo
  • Lower memory overhead per keyMenor uso de memoria por clave
  • No persistence (pure in-memory)Sin persistencia (puramente en memoria)
  • Simpler operational modelModelo operativo más simple
  • Client-side sharding via consistent hashingSharding del lado del cliente vía hashing consistente

Best for: simple object caching, HTML fragment caching, and scenarios where the cache is purely ephemeral. When you only need to store and retrieve serialized blobs and want maximum simplicity with minimal memory overhead.Ideal para: caché simple de objetos, caché de fragmentos HTML y escenarios donde el caché es puramente efímero. Cuando solo necesitas almacenar y recuperar blobs serializados con mínima complejidad.

Redis is the default choice for most modern applicationsRedis es la opción predeterminada para la mayoría de aplicaciones modernas

Multi-Tier Caching ArchitectureArquitectura de Caché Multi-Nivel

CDN for static assets, application cache for dynamic data. Each layer absorbs traffic before it reaches the next.CDN para activos estáticos, caché de aplicación para datos dinámicos. Cada capa absorbe tráfico antes de que llegue a la siguiente.

When to UseCuándo Usar

High-traffic applications with mixed static and dynamic contentAplicaciones de alto tráfico con contenido estático y dinámico mixto

The CDN handles images, CSS, JavaScript, and other static assets at the edge, close to users. The application-level cache (Redis) stores frequently accessed database results, API responses, and session data. Together, they reduce origin server load by 90%+.El CDN maneja imágenes, CSS, JavaScript y otros activos estáticos en el borde, cerca de los usuarios. El caché a nivel de aplicación (Redis) almacena resultados de BD frecuentemente accedidos, respuestas de API y datos de sesión. Juntos, reducen la carga del servidor origen en 90%+.

Technology ChoiceElección de Tecnología

CDN: CloudFront, Cloudflare. App cache: Redis, Memcached.CDN: CloudFront, Cloudflare. App caché: Redis, Memcached.

CloudFront integrates natively with AWS services (S3, ALB, API Gateway). Cloudflare offers a simpler setup with DDoS protection included. For the application layer, Redis is preferred for its rich data structures, while Memcached is simpler for pure key-value caching.CloudFront se integra nativamente con servicios AWS (S3, ALB, API Gateway). Cloudflare ofrece una configuración más simple con protección DDoS incluida. Para la capa de aplicación, Redis es preferido por sus estructuras de datos ricas, mientras que Memcached es más simple para caché puro clave-valor.
