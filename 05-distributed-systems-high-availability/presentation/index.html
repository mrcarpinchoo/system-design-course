What Are Distributed Systems?¿Qué Son los Sistemas Distribuidos?

A collection of independent computers that appear as a single system to users.Una colección de computadoras independientes que aparecen como un solo sistema para los usuarios.

Key InsightPunto Clave

Nodes communicate over a network, each with its own local memory and clock.Los nodos se comunican a través de la red, cada uno con su propia memoria y reloj local.

Unlike a single machine where components share memory and a system clock, distributed systems must handle message passing, network latency, and the absence of a global time reference. This fundamental difference drives most of the complexity in distributed computing.A diferencia de una sola máquina donde los componentes comparten memoria y un reloj de sistema, los sistemas distribuidos deben manejar paso de mensajes, latencia de red y la ausencia de una referencia temporal global. Esta diferencia fundamental impulsa la mayoría de la complejidad en la computación distribuida.

Key CharacteristicsCaracterísticas Clave

ScalabilityEscalabilidad

Add more nodes to handle growing workloads without redesigning the system.Agrega más nodos para manejar cargas de trabajo crecientes sin rediseñar el sistema.

Horizontal scaling distributes load across machines. Vertical scaling has hardware limits, but horizontal scaling can grow almost indefinitely. This is why services like Netflix and Google rely on distributed architectures.El escalado horizontal distribuye la carga entre máquinas. El escalado vertical tiene límites de hardware, pero el horizontal puede crecer casi indefinidamente. Por eso servicios como Netflix y Google dependen de arquitecturas distribuidas.

ReliabilityConfiabilidad

No single point of failure — if one node fails, others continue serving.Sin punto único de falla — si un nodo falla, los demás continúan sirviendo.

Redundancy across nodes means the system can tolerate individual failures. Data replication ensures no data loss. However, reliability requires careful design of failure detection and recovery mechanisms.La redundancia entre nodos significa que el sistema puede tolerar fallos individuales. La replicación de datos asegura que no haya pérdida. Sin embargo, la confiabilidad requiere un diseño cuidadoso de detección y recuperación de fallos.

AutonomyAutonomía

Each node operates independently, making local decisions without a central coordinator.Cada nodo opera de manera independiente, tomando decisiones locales sin un coordinador central.

Autonomous nodes can process requests even when disconnected from other parts of the system. This enables geographic distribution and resilience to network partitions, but introduces consistency challenges.Los nodos autónomos pueden procesar solicitudes incluso cuando están desconectados de otras partes del sistema. Esto permite la distribución geográfica y resiliencia a particiones de red, pero introduce desafíos de consistencia.

No Shared Memory/ClockSin Memoria/Reloj Compartido

Nodes have separate address spaces and unsynchronized clocks.Los nodos tienen espacios de memoria separados y relojes no sincronizados.

This means you cannot assume operations on different nodes happen in a particular order. Techniques like vector clocks, Lamport timestamps, and consensus algorithms (Paxos, Raft) address ordering and agreement challenges.Esto significa que no puedes asumir que las operaciones en diferentes nodos ocurren en un orden particular. Técnicas como relojes vectoriales, timestamps de Lamport y algoritmos de consenso (Paxos, Raft) abordan los desafíos de ordenamiento y acuerdo.

When to use which?¿Cuándo usar cuál?

Choose distributed for availability and geographic reach; parallel for compute-intensive tasks on a single dataset.Elige distribuido para disponibilidad y alcance geográfico; paralelo para tareas de cómputo intensivo sobre un solo dataset.

In practice, modern systems combine both: a distributed cluster of machines where each machine uses parallel processing (multi-core CPUs, GPUs). Understanding the distinction helps you reason about failure modes and consistency guarantees.En la práctica, los sistemas modernos combinan ambos: un cluster distribuido de máquinas donde cada máquina usa procesamiento paralelo (CPUs multi-core, GPUs). Entender la distinción ayuda a razonar sobre modos de fallo y garantías de consistencia.

ScalabilityEscalabilidad

Add nodes to grow capacity linearly.Agrega nodos para crecer la capacidad linealmente.

Unlike vertical scaling (bigger machines), horizontal scaling has no theoretical upper limit. Cloud providers make it easy to add instances on demand, paying only for what you use.A diferencia del escalado vertical (máquinas más grandes), el escalado horizontal no tiene límite teórico superior. Los proveedores cloud facilitan agregar instancias bajo demanda, pagando solo por lo que usas.

ReliabilityConfiabilidad

Replicate data across nodes for fault tolerance.Replica datos entre nodos para tolerancia a fallos.

If one replica fails, others serve the data. Techniques like quorum reads and writes ensure consistency even during failures. The key trade-off is between consistency and availability (CAP theorem).Si una réplica falla, las demás sirven los datos. Técnicas como lecturas y escrituras de quórum aseguran consistencia durante fallos. La compensación clave es entre consistencia y disponibilidad (teorema CAP).

PerformanceRendimiento

Distribute work to reduce latency and increase throughput.Distribuye trabajo para reducir latencia y aumentar el throughput.

Geographic distribution places data closer to users, reducing round-trip times. Partitioning (sharding) allows parallel processing of queries across different subsets of data.La distribución geográfica coloca datos cerca de los usuarios, reduciendo tiempos de ida y vuelta. El particionamiento (sharding) permite procesamiento paralelo de consultas en diferentes subconjuntos de datos.

ChallengesDesafíos

Network FailuresFallos de Red

Messages can be lost, delayed, duplicated, or delivered out of order.Los mensajes pueden perderse, retrasarse, duplicarse o entregarse fuera de orden.

The “Two Generals Problem” proves that reliable communication over unreliable networks is impossible to guarantee. Real systems use timeouts, retries with idempotency, and acknowledgments to mitigate this.El “Problema de los Dos Generales” demuestra que la comunicación confiable sobre redes no confiables es imposible de garantizar. Los sistemas reales usan timeouts, reintentos con idempotencia y acknowledgments para mitigar esto.

ConsistencyConsistencia

Keeping all replicas in sync is expensive and sometimes impossible (CAP theorem).Mantener todas las réplicas sincronizadas es costoso y a veces imposible (teorema CAP).

Strong consistency requires coordination (two-phase commit, consensus), which increases latency. Eventual consistency is faster but allows temporary stale reads. Choose based on your application’s tolerance for stale data.La consistencia fuerte requiere coordinación (two-phase commit, consenso), lo que aumenta la latencia. La consistencia eventual es más rápida pero permite lecturas temporalmente desactualizadas. Elige según la tolerancia de tu aplicación a datos desactualizados.

Partial FailuresFallos Parciales

Some nodes may fail while others continue — hard to detect and handle.Algunos nodos pueden fallar mientras otros continúan — difícil de detectar y manejar.

A slow node is harder to handle than a dead node — is it processing or has it crashed? Heartbeats, timeouts, and failure detectors (phi accrual) help distinguish between slow and dead nodes.Un nodo lento es más difícil de manejar que uno muerto — ¿está procesando o se cayó? Heartbeats, timeouts y detectores de fallos (phi accrual) ayudan a distinguir entre nodos lentos y muertos.

Clock SynchronizationSincronización de Relojes

Different machines have different clock drift, making ordering events difficult.Diferentes máquinas tienen diferente desviación de reloj, dificultando el ordenamiento de eventos.

NTP can synchronize clocks to within milliseconds, but that’s often not enough. Google Spanner uses GPS and atomic clocks (TrueTime API) to bound clock uncertainty. Most systems use logical clocks instead of wall clocks for ordering.NTP puede sincronizar relojes con precisión de milisegundos, pero frecuentemente no es suficiente. Google Spanner usa GPS y relojes atómicos (TrueTime API) para acotar la incertidumbre del reloj. La mayoría de los sistemas usan relojes lógicos en lugar de relojes de pared para ordenamiento.

What Is High Availability?¿Qué Es Alta Disponibilidad?

A system’s ability to remain operational and accessible for a very high percentage of time.La capacidad de un sistema para permanecer operativo y accesible durante un porcentaje muy alto del tiempo.

99.999% “Five nines” uptime target Objetivo de uptime “cinco nueves”
5m 15s Max downtime per year (5 nines) Tiempo de inactividad máx/año (5 nueves)
$$$ Each additional nine costs more Cada nueve adicional cuesta más

Why does it matter?¿Por qué importa?

Downtime costs money, trust, and users. SLAs define availability commitments.El tiempo de inactividad cuesta dinero, confianza y usuarios. Los SLAs definen compromisos de disponibilidad.

Amazon estimated that every 100ms of added latency costs 1% in sales. Google found an extra 500ms in search results caused 20% traffic drop. Availability is measured as: Availability = Uptime / (Uptime + Downtime). SLAs often include financial penalties for breaches.Amazon estimó que cada 100ms de latencia adicional cuesta 1% en ventas. Google encontró que 500ms extra en resultados de búsqueda causó 20% de caída en tráfico. La disponibilidad se mide como: Disponibilidad = Uptime / (Uptime + Downtime). Los SLAs frecuentemente incluyen penalizaciones financieras por incumplimiento.

MTBF

Mean Time Between Failures — how long the system runs before failing again.Tiempo Medio Entre Fallos — cuánto funciona el sistema antes de fallar de nuevo.

MTTR

Mean Time To Repair — how long it takes to restore the system after a failure.Tiempo Medio de Reparación — cuánto tarda en restaurar el sistema después de un fallo.

Redundancy: Hot-ColdRedundancia: Hot-Cold

Hot-Cold Trade-offsCompensaciones Hot-Cold

Cheapest redundancy model, but slowest failover (minutes).Modelo de redundancia más económico, pero failover más lento (minutos).

The backup is not running — it must boot up, load state, and warm caches before serving traffic. This can take minutes. Suitable for systems that can tolerate brief outages (e.g., batch processing, internal tools). Data loss depends on replication lag (RPO).El respaldo no está ejecutándose — debe arrancar, cargar estado y calentar cachés antes de servir tráfico. Esto puede tomar minutos. Adecuado para sistemas que toleran interrupciones breves (ej., procesamiento por lotes, herramientas internas). La pérdida de datos depende del retraso de replicación (RPO).

Redundancy: Hot-Warm & Hot-HotRedundancia: Hot-Warm y Hot-Hot

Hot-Warm

Hot-Hot

Hot-WarmHot-Warm

Backup is running and receiving data but only handles reads.El respaldo está activo y recibe datos pero solo maneja lecturas.

Failover is faster because the warm standby already has state loaded and connections established. It only needs to be promoted to primary (accept writes). Common in database replicas: read replicas offload read traffic and can be promoted on failure.El failover es más rápido porque el standby ya tiene el estado cargado y las conexiones establecidas. Solo necesita promoverse a primario (aceptar escrituras). Común en réplicas de BD: las réplicas de lectura descargan tráfico de lectura y pueden promoverse ante un fallo.

Hot-Hot (Active-Active)Hot-Hot (Activo-Activo)

Both nodes fully active, sharing the load. No failover delay.Ambos nodos completamente activos, compartiendo la carga. Sin retraso de failover.

Provides the highest availability but is the most complex and expensive. Requires conflict resolution for concurrent writes (last-writer-wins, CRDTs, or application-level merge). Used by global services that need zero-downtime deployments and geographic load balancing.Proporciona la mayor disponibilidad pero es el más complejo y costoso. Requiere resolución de conflictos para escrituras concurrentes (last-writer-wins, CRDTs, o merge a nivel de aplicación). Usado por servicios globales que necesitan despliegues sin tiempo de inactividad y balanceo de carga geográfico.

Resiliency TechniquesTécnicas de Resiliencia

TimeoutsTimeouts

Set deadlines for operations to avoid waiting indefinitely.Establece plazos para operaciones y evita esperar indefinidamente.

Without timeouts, a slow downstream service can exhaust your thread pool and cascade failures. Set aggressive timeouts and handle them gracefully (return cached data, degrade functionality).Sin timeouts, un servicio lento puede agotar tu pool de hilos y provocar fallos en cascada. Establece timeouts agresivos y manéjalos adecuadamente (retorna datos en caché, degrada funcionalidad).

RetriesReintentos

Automatically retry failed requests with exponential backoff.Reintenta automáticamente solicitudes fallidas con backoff exponencial.

Use exponential backoff with jitter to avoid thundering herd. Ensure operations are idempotent before retrying. Limit maximum retries to prevent infinite loops. Add retry budgets to limit total retry traffic.Usa backoff exponencial con jitter para evitar avalanchas. Asegura que las operaciones sean idempotentes antes de reintentar. Limita los reintentos máximos para prevenir loops infinitos. Agrega presupuestos de reintento para limitar el tráfico total.

Circuit BreakersCircuit Breakers

Stop calling a failing service to let it recover.Deja de llamar a un servicio fallando para permitir su recuperación.

Three states: Closed (normal), Open (failing — reject immediately), Half-Open (test with limited requests). Prevents cascading failures by failing fast. Netflix Hystrix popularized this pattern; now built into service meshes like Istio.Tres estados: Cerrado (normal), Abierto (fallando — rechaza inmediatamente), Semi-Abierto (prueba con peticiones limitadas). Previene fallos en cascada al fallar rápidamente. Netflix Hystrix popularizó este patrón; ahora integrado en service meshes como Istio.

BulkheadBulkhead

Isolate components so one failure does not sink the whole system.Aísla componentes para que un fallo no hunda todo el sistema.

Named after ship bulkheads that contain flooding. Implement with separate thread pools, connection pools, or processes per dependency. If one dependency is slow, it only exhausts its own pool, not the shared resources.Nombrado por los mamparos de barcos que contienen inundaciones. Implementa con pools de hilos, pools de conexiones o procesos separados por dependencia. Si una dependencia es lenta, solo agota su propio pool, no los recursos compartidos.

Load SheddingDescarga de Carga

Reject excess requests to protect the system under heavy load.Rechaza solicitudes excesivas para proteger el sistema bajo carga pesada.

Better to serve some requests well than all requests poorly. Prioritize critical traffic (e.g., payments over analytics). Return HTTP 503 with Retry-After header. Cloud load balancers can shed traffic automatically based on backend health.Es mejor servir algunas solicitudes bien que todas mal. Prioriza tráfico crítico (ej., pagos sobre analítica). Retorna HTTP 503 con header Retry-After. Los balanceadores cloud pueden descargar tráfico automáticamente basado en la salud del backend.

Health ChecksHealth Checks

Periodically verify that each component is healthy and responsive.Verifica periódicamente que cada componente esté saludable y respondiendo.

Liveness checks (is it running?) vs readiness checks (can it serve traffic?). Shallow checks ping the process; deep checks verify database connections, disk space, and dependent services. Kubernetes uses both for pod lifecycle management.Checks de vida (está corriendo?) vs checks de preparación (puede servir tráfico?). Checks superficiales hacen ping al proceso; checks profundos verifican conexiones a BD, espacio en disco y servicios dependientes. Kubernetes usa ambos para gestión del ciclo de vida de pods.

SummaryResumen

Distributed SystemsSistemas Distribuidos

Independent nodes communicating over a network, providing scalability and reliability at the cost of complexity.Nodos independientes comunicados por red, proporcionando escalabilidad y confiabilidad al costo de complejidad.

High AvailabilityAlta Disponibilidad

Measured in nines (99.9% to 99.9999%). Each additional nine requires exponentially more investment in redundancy and automation.Medida en nueves (99.9% a 99.9999%). Cada nueve adicional requiere inversión exponencialmente mayor en redundancia y automatización.

ResiliencyResiliencia

Combine redundancy (hot-cold, hot-warm, hot-hot) with resilience patterns (timeouts, retries, circuit breakers) to build robust systems.Combina redundancia (hot-cold, hot-warm, hot-hot) con patrones de resiliencia (timeouts, reintentos, circuit breakers) para construir sistemas robustos.
