Layer 4 vs Layer 7Capa 4 vs Capa 7

L4 Use CasesCasos de uso L4

High-throughput TCP proxying, database connections, gaming servers.Proxy TCP de alto rendimiento, conexiones a bases de datos, servidores de juegos.

L4 load balancers operate at the transport layer. They make routing decisions based on IP address and TCP/UDP port without reading the application payload. This makes them extremely fast but unable to make content-aware routing decisions.Los balanceadores L4 operan en la capa de transporte. Toman decisiones de enrutamiento basadas en dirección IP y puerto TCP/UDP sin leer el contenido de la aplicación. Esto los hace extremadamente rápidos pero incapaces de tomar decisiones basadas en contenido.

L7 Use CasesCasos de uso L7

URL-based routing, A/B testing, SSL termination, API gateways.Enrutamiento por URL, pruebas A/B, terminación SSL, API gateways.

L7 load balancers read the full HTTP request including URL path, headers, and cookies. This enables content-based routing (e.g., /api to backend A, /images to backend B), SSL termination, request transformation, and Web Application Firewall (WAF) integration.Los balanceadores L7 leen la solicitud HTTP completa incluyendo ruta URL, headers y cookies. Esto permite enrutamiento basado en contenido (ej., /api al backend A, /images al backend B), terminación SSL, transformación de solicitudes e integración con WAF.

Round RobinRound Robin

How it worksCómo funciona

Each request is assigned to the next server in a circular sequence: 1 → 2 → 3 → 1 → ...Cada solicitud se asigna al siguiente servidor en secuencia circular: 1 → 2 → 3 → 1 → ...

Round Robin is the simplest algorithm. It works best when all servers have equal capacity and requests have similar processing costs. It does not account for server load or connection count, which can lead to uneven distribution with heterogeneous workloads.Round Robin es el algoritmo más simple. Funciona mejor cuando todos los servidores tienen la misma capacidad y las solicitudes tienen costos de procesamiento similares. No considera la carga del servidor ni el número de conexiones, lo que puede generar distribución desigual con cargas heterogéneas.

Weighted Round RobinRound Robin Ponderado

When to use weightsCuándo usar pesos

Assign proportional weights when servers have different CPU, memory, or network capacity.Asigna pesos proporcionales cuando los servidores tienen diferente CPU, memoria o capacidad de red.

Weighted Round Robin is ideal during rolling upgrades (send less traffic to new instances), mixed hardware environments, or canary deployments where a small percentage of traffic goes to a new version. Weights can be adjusted dynamically in most load balancers.El Round Robin ponderado es ideal durante actualizaciones progresivas (enviar menos tráfico a nuevas instancias), entornos con hardware mixto, o despliegues canary donde un pequeño porcentaje del tráfico va a una nueva versión. Los pesos se pueden ajustar dinámicamente en la mayoría de los balanceadores.

Least ConnectionsMenor Número de Conexiones

Best for variable workloadsMejor para cargas variables

Routes new requests to the server with the fewest active connections.Enruta nuevas solicitudes al servidor con menos conexiones activas.

Least Connections excels when requests have varying processing times (e.g., some queries take 10ms, others 5s). Unlike Round Robin, it naturally adapts to server capacity differences and prevents slow servers from accumulating a backlog of requests.Least Connections sobresale cuando las solicitudes tienen tiempos de procesamiento variables (ej., algunas consultas toman 10ms, otras 5s). A diferencia de Round Robin, se adapta naturalmente a las diferencias de capacidad y evita que los servidores lentos acumulen solicitudes pendientes.

Hash-Based RoutingEnrutamiento Basado en Hash

Session affinity via hashingAfinidad de sesión vía hashing

The same client IP always maps to the same server, enabling sticky sessions without cookies.La misma IP de cliente siempre se mapea al mismo servidor, habilitando sesiones fijas sin cookies.

Hash-based routing is essential for stateful applications (shopping carts, WebSocket connections) and caching layers where cache locality matters. Consistent hashing minimizes disruption when servers are added or removed by only reassigning a fraction of the keys.El enrutamiento basado en hash es esencial para aplicaciones con estado (carritos de compra, conexiones WebSocket) y capas de caché donde la localidad importa. El hashing consistente minimiza la interrupción cuando se agregan o eliminan servidores, reasignando solo una fracción de las claves.

Health ChecksVerificaciones de Salud

Automatic failure detectionDetección automática de fallos

The LB periodically pings each server. Unhealthy servers are removed from the pool until they recover.El LB hace ping periódicamente a cada servidor. Los servidores no saludables se eliminan del pool hasta que se recuperan.

Health checks can be passive (monitoring response codes from real traffic) or active (dedicated HTTP/TCP probes). Configure thresholds: unhealthy after N consecutive failures, healthy after M consecutive successes. Common endpoints: /health, /ready, /live for Kubernetes readiness and liveness probes.Las verificaciones pueden ser pasivas (monitoreando códigos de respuesta del tráfico real) o activas (sondas HTTP/TCP dedicadas). Configura umbrales: no saludable después de N fallos consecutivos, saludable después de M éxitos consecutivos. Endpoints comunes: /health, /ready, /live para sondas de Kubernetes.

Scaling the Load BalancerEscalando el Balanceador

Active-PassiveActivo-Pasivo

One LB handles traffic; the standby takes over on failure via a floating IP (VIP).Un LB maneja el tráfico; el respaldo toma el control ante una falla mediante una IP flotante (VIP).

Active-Passive (also called Primary-Standby) uses protocols like VRRP or keepalived to detect failures and reassign the floating IP within seconds. Simple but underutilizes the passive node.Activo-Pasivo (también llamado Primario-Espera) usa protocolos como VRRP o keepalived para detectar fallos y reasignar la IP flotante en segundos. Simple pero subutiliza el nodo pasivo.

Active-ActiveActivo-Activo

Both LBs serve traffic simultaneously using DNS round robin or Anycast.Ambos LBs sirven tráfico simultáneamente usando DNS round robin o Anycast.

Active-Active maximizes utilization and provides higher throughput. DNS-based approaches distribute clients across LBs. Anycast assigns the same IP to multiple LBs and relies on BGP routing to direct traffic to the nearest one. More complex but no wasted capacity.Activo-Activo maximiza la utilización y proporciona mayor rendimiento. Los enfoques basados en DNS distribuyen clientes entre LBs. Anycast asigna la misma IP a múltiples LBs y usa enrutamiento BGP para dirigir tráfico al más cercano. Más complejo pero sin capacidad desperdiciada.

Load Balancing PlatformsPlataformas de Balanceo

AWS ALB / NLB

ALB (L7) for HTTP/HTTPS routing; NLB (L4) for ultra-low latency TCP/UDP.ALB (L7) para enrutamiento HTTP/HTTPS; NLB (L4) para TCP/UDP de latencia ultra baja.

ALB supports path-based and host-based routing, WebSocket, gRPC, and integrates with WAF and Cognito. NLB handles millions of requests per second with static IPs. Both integrate with Auto Scaling and ECS/EKS.ALB soporta enrutamiento por ruta y host, WebSocket, gRPC, e integra con WAF y Cognito. NLB maneja millones de solicitudes por segundo con IPs estáticas. Ambos integran con Auto Scaling y ECS/EKS.

Azure Load Balancer

L4 load balancer with zone-redundant frontends and Application Gateway for L7.Balanceador L4 con frontends redundantes por zona y Application Gateway para L7.

Azure Load Balancer provides high availability with availability zones. Application Gateway adds URL routing, SSL offloading, and WAF. Traffic Manager provides DNS-based global load balancing across regions.Azure Load Balancer proporciona alta disponibilidad con zonas de disponibilidad. Application Gateway agrega enrutamiento por URL, descarga SSL y WAF. Traffic Manager ofrece balanceo global basado en DNS entre regiones.

GCP Cloud Load Balancing

Global, Anycast-based LB with automatic multi-region failover.LB global basado en Anycast con failover multiregión automático.

GCP offers a single global IP that routes to the nearest healthy backend using Google's backbone network. Supports HTTP(S), TCP/SSL proxy, and network load balancing. Cloud Armor provides DDoS protection and WAF capabilities.GCP ofrece una IP global única que enruta al backend saludable más cercano usando la red troncal de Google. Soporta HTTP(S), proxy TCP/SSL y balanceo de red. Cloud Armor proporciona protección DDoS y capacidades WAF.

HAProxy / Nginx

Open-source L4/L7 load balancers with full configuration control.Balanceadores L4/L7 de código abierto con control total de configuración.

HAProxy is the industry standard for high-performance TCP/HTTP load balancing. Nginx provides reverse proxy, caching, and LB in one package. Both offer health checks, sticky sessions, SSL termination, and support millions of concurrent connections. Ideal for on-premise or hybrid cloud deployments.HAProxy es el estándar de la industria para balanceo TCP/HTTP de alto rendimiento. Nginx proporciona proxy inverso, caché y LB en un paquete. Ambos ofrecen health checks, sesiones fijas, terminación SSL y soportan millones de conexiones concurrentes. Ideales para despliegues on-premise o nube híbrida.
